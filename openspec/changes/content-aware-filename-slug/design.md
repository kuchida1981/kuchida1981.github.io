## Context

`scripts/generate_daily_post.py` が毎日 Gemini API で記事を生成し、`content/posts/YYYY/MM/YYYY-mm-dd-daily-news.md` に保存している。記事タイトルは毎回ユニークだが、ファイル名は固定の `daily-news` で内容を反映していない。

現在の処理フロー:
1. RSS フィードから記事取得
2. Gemini API で記事生成（1回目）
3. front matter から title 抽出
4. `YYYY-mm-dd-daily-news.md` として保存

## Goals / Non-Goals

**Goals:**
- ファイル名を内容に応じた英語スラッグにする
- 記事生成プロンプトに影響を与えない
- スラッグ生成の失敗が記事生成を妨げない

**Non-Goals:**
- 既存記事のファイル名変更
- front matter の `slug` フィールド追加（Hugo の URL 上書きは使わない）
- 記事生成プロンプトの変更

## Decisions

### D1: スラッグ生成を独立した2回目の API 呼び出しにする

記事生成後に title を抽出し、別の Gemini API 呼び出しで英語スラッグを生成する。

**理由**: 記事生成プロンプトを変更しないことで、記事品質への影響をゼロにする。スラッグ生成は入出力ともに短く、コストは無視できるレベル。

**却下した代替案**:
- 案C-1（プロンプト内でslugも生成）: プロンプトが複雑化し、記事品質とスラッグ品質が連動してブレる
- 案A（titleから機械的に変換）: 日本語→英語変換の精度問題。追加ライブラリが必要

### D2: スラッグのサニタイズとバリデーションを Python 側で行う

Gemini の出力を信頼せず、以下の後処理を行う:
1. 前後の空白・改行を除去
2. ASCII 英数字とハイフン以外を除去
3. 連続ハイフンを単一ハイフンに統合
4. 先頭・末尾のハイフンを除去
5. 6語（ハイフン区切り）を超える場合は先頭6語で切る
6. 結果が空文字なら `daily-news` にフォールバック

### D3: フォールバックスラッグとして `daily-news` を使う

スラッグ生成の API 呼び出し自体が例外を投げた場合も、`daily-news` にフォールバックして記事保存を続行する。

## Risks / Trade-offs

- **[スラッグの品質が不安定]** → サニタイズ + バリデーションで最低限の品質を保証。最悪でも `daily-news` にフォールバック
- **[API コスト微増]** → スラッグ生成は入出力合わせて100トークン未満。1日1回なので月額への影響は無視できる
- **[同日2回実行時のファイル名衝突]** → 同一タイトルなら同一スラッグになり上書き（現状と同じ挙動）。異なるタイトルなら異なるスラッグで共存する。現状でも上書きなのでリスクは増えない
